# WayWeaver AI 产品需求文档

文档状态：Draft  
版本：0.1.0  
阶段：T001  
最后更新：2026-07-23

## 1. 产品愿景

WayWeaver AI 希望解决传统旅行规划中信息分散、约束复杂、计划不可执行和修改成本高的问题。

系统不只是生成一段看起来合理的文字，而是输出：

- 时间上可执行；
- 路线上可到达；
- 营业时间内可访问；
- 基本符合预算；
- 适应用户体力与兴趣；
- 可以追溯信息来源；
- 可以继续修改和回滚；
- 能明确说明不确定性的旅行计划。

## 2. 产品原则

### 2.1 事实与建议分离

- 天气、路线、营业时间、航班和酒店来自实时工具或带时间戳的缓存。
- LLM 不得凭记忆声称某地点当前营业、某航班存在或某价格准确。
- 系统估算必须标记为“估算”。
- 实时结果必须记录 Provider、获取时间和有效期。

### 2.2 LLM 与程序职责分离

LLM 负责：

- 理解自然语言；
- 提取结构化需求；
- 识别缺失信息；
- 决定调用哪些工具；
- 理解用户的修改意图；
- 解释行程安排；
- 生成自然语言总结。

确定性程序负责：

- 日期与时区计算；
- 时间冲突检查；
- 路线距离和耗时处理；
- 营业时间检查；
- 预算加总；
- 行程排序；
- 状态转换；
- 权限校验；
- 版本控制。

外部工具负责：

- 地点与餐厅信息；
- 路线和距离矩阵；
- 天气；
- 航班；
- 酒店；
- 汇率。

### 2.3 用户始终拥有最终控制权

- 系统不能未经确认完成支付或预订。
- 重要修改应展示变更摘要。
- 危险、昂贵或不可逆操作需要显式确认。
- 用户可以查看历史版本并恢复。

## 3. 目标用户

### 3.1 核心用户

计划 2～7 天单城市自由行的个人、情侣或家庭用户。

### 3.2 典型用户画像

#### 用户 A：第一次自由行

- 不熟悉目的地；
- 希望快速得到合理的每日安排；
- 需要交通、天气和预算提示；
- 容易安排过多景点。

#### 用户 B：家庭旅行

- 有儿童或老人同行；
- 对步行距离、休息时间和用餐时间敏感；
- 更看重可执行性而不是景点数量。

#### 用户 C：深度兴趣旅行者

- 对历史、美食、摄影或自然景观有明确兴趣；
- 有必去和不想去的地点；
- 希望反复调整并比较不同版本。

## 4. 第一版范围

### 4.1 默认范围

| 项目 | 第一版决定 |
|---|---|
| 平台 | Web |
| 界面语言 | 中文 |
| 旅行类型 | 单城市、多日自由行 |
| 地图范围 | 中国优先，Provider 接口支持全球扩展 |
| 用户类型 | 个人用户 |
| 预订能力 | 搜索和推荐，不支付、不出票 |
| 规划算法 | 规则、评分和贪心算法 |
| Agent 形态 | 单一受控工作流，不采用多 Agent |
| 前端优先级 | 后端闭环完成后开发 |

### 4.2 不在第一版范围

- 真实支付、出票和酒店下单；
- 旅行社订单管理；
- 企业差旅审批；
- 多城市跨国复杂路线；
- 签证结论或法律建议；
- 自动操作用户邮箱；
- Kubernetes；
- 微服务拆分；
- 自训练大模型；
- 大规模社交推荐；
- 完整离线地图。

## 5. 核心用户故事

### US-001 创建旅行

作为用户，我希望输入出发地、目的地、日期、人数和预算，创建一个旅行项目。

验收：

- 必填字段缺失时返回明确错误；
- 开始日期不得晚于结束日期；
- 结束日期不得早于当前允许范围；
- 人数必须大于零；
- 预算必须为非负数；
- 创建成功后返回 Trip ID。

### US-002 自然语言描述旅行

作为用户，我希望用自然语言描述旅行，而不是填写所有表单。

验收：

- 系统将文本转换为结构化需求；
- 结构化结果可以由用户确认和修改；
- 缺失关键字段时向用户提问；
- 系统不得猜测日期、人数和目的地。

### US-003 保存旅行偏好

作为用户，我希望保存兴趣、旅行节奏、步行强度和饮食需求。

验收：

- 支持兴趣标签；
- 支持必去与避开地点；
- 支持慢节奏、正常、紧凑；
- 支持儿童、老人和无障碍约束；
- 偏好变化能够触发行程重新评估。

### US-004 生成多日行程

作为用户，我希望系统生成按天、按时间排列的行程。

验收：

- 每日有开始和结束时间；
- 活动间包含交通时间；
- 安排午餐、晚餐或可解释的用餐空档；
- 必去地点被优先安排；
- 超出时间窗口时不会强行塞入；
- 返回规划说明和来源。

### US-005 检查行程可行性

作为用户，我希望系统发现不合理的计划。

验收：

- 检测时间重叠；
- 检测营业时间冲突；
- 检测交通时间不足；
- 检测每日步行强度；
- 检测预算超限；
- 检测必去地点遗漏；
- 对无法解决的冲突给出明确说明。

### US-006 根据天气调整

作为用户，我希望室外活动避开明显不利天气。

验收：

- 行程日期在预报范围内时查询天气；
- 降雨时优先安排室内地点；
- 高温、低温或强风给出提醒；
- 日期超出可靠预报范围时不输出伪实时天气；
- 保存天气数据的获取时间。

### US-007 对话修改

作为用户，我希望通过自然语言修改当前行程。

示例：

```text
第二天太累了，减少一个景点。
第三天下雨的话改成室内活动。
把酒店预算调整到每晚 600 元。
必须安排兵马俑。
```

验收：

- 系统提取结构化变更；
- 不修改无关日期；
- 修改后重新执行冲突检查；
- 创建新版本；
- 返回变更摘要。

### US-008 查看和恢复版本

作为用户，我希望查看历史行程并恢复旧版本。

验收：

- 每次生成或修改产生新版本；
- 旧版本不可被直接覆盖；
- 可以比较两个版本；
- 可以选择一个历史版本作为当前版本。

### US-009 查看预算

作为用户，我希望查看旅行总预算和分类预算。

预算分类：

- 城际交通；
- 住宿；
- 门票；
- 餐饮；
- 市内交通；
- 购物；
- 应急预算。

验收：

- 显示货币单位；
- 区分实时价格和估算价格；
- 显示价格获取时间；
- 超预算时提供可解释的调整建议。

### US-010 查看信息来源

作为用户，我希望知道天气、路线、地点和价格从哪里获得。

验收：

- 外部事实包含 Provider；
- 包含 `retrieved_at`；
- 必要时包含 `expires_at`；
- 回答能引用对应工具结果；
- 缓存过期后不作为实时数据展示。

## 7. 非功能需求

### 性能

- 普通 CRUD API 的开发环境目标 P95 小于 500ms；
- 外部 Provider 必须设置连接和总超时；
- 长时间规划任务使用后台执行并通过事件流返回进度；
- Provider 请求应支持缓存和并发限制。

### 可靠性

- 外部工具失败不能导致整个 API 进程崩溃；
- 可恢复错误支持有限次数重试；
- Agent 必须有最大轮数和总超时；
- 行程保存使用事务；
- 重复创建请求支持幂等键。

### 安全

- 密码只保存哈希；
- JWT 或会话密钥不写入日志；
- 所有 Trip 数据校验所有权；
- 外部工具只允许白名单；
- 不允许 LLM 直接执行任意代码；
- 对 Prompt Injection 进行边界隔离；
- API Key 仅从环境变量或密钥服务读取。

### 可观测性

- 每个请求具有 `request_id`；
- 每次 Agent 运行具有 `run_id`；
- 每次 Tool Call 记录名称、参数摘要、状态和耗时；
- 记录 Token 使用量和估算成本；
- 记录 Provider 错误、重试和缓存命中。

### 可测试性

- 所有外部 Provider 都有 Fake 实现；
- 单元测试不得依赖真实网络；
- API 测试使用独立数据库；
- 规划算法能够使用固定输入重复测试；
- Agent 评测集纳入版本控制。

## 8. 成功指标

### 工程指标

- 核心业务测试覆盖率不低于 80%；
- CI 中所有测试、格式和静态检查通过；
- 全新环境可通过 Docker Compose 启动；
- Git 历史不包含真实密钥。

### 产品指标

- 结构化需求提取字段准确率；
- 必去地点覆盖率；
- 时间冲突率；
- 营业时间冲突率；
- 预算超限率；
- 工具选择正确率；
- 行程生成成功率；
- 用户修改后无关行程项保持率。

### 演示验收场景

```text
我和父母计划国庆去西安 4 天，从上海出发，总预算 12000 元。
父母不适合长时间步行，我们喜欢历史文化，必须去兵马俑，
希望住宿靠近地铁，行程不要太赶。
```

最终演示必须展示：

1. 需求提取；
2. 缺失信息确认；
3. 地点、路线和天气工具调用；
4. 多日行程；
5. 预算；
6. 冲突检查；
7. 来源；
8. 对话修改；
9. 新版本；
10. Agent 运行追踪。
